9주차 · 3장 계획의 이해 / 문제정의 / 타당성분석

계획의 이해

  • 소프트웨어 개발계획 = 비용·기간·자원 계획. 계획이 없으면 → 일정지연·비용초과·품질저하·유지보수비 증가
  • 계획 수립 절차: 문제정의 → 타당성분석 → 비용산정 → 일정계획 → 위험분석

문제정의 (개발의 첫 작업)

  • 무엇을 개발할지 명확히 정의하고 개발 범위 결정 → 초기 타당성·계획의 기초자료
  • 필요사항: 분야 배경지식, 현행 시스템 사용, 실무 담당자 면담을 통한 자료수집·분석

타당성분석 3종

  • 경제적: 투자효율성(비용편익분석 CBA), 시장성 확인 → 개발여부 판단
  • 기술적: 현재 기술로 구현 가능한지, HW 성능, 개발자 기술력 검토
  • 법적: SW·도구 사용의 법적 문제(지적소유권·프로그램보호법) 검토

개발비에 영향을 주는 요소

  • 프로그래머 자질(초급 vs 고급 생산성 차이), 소프트웨어 복잡도·크기, 가용시간, 요구 신뢰도, 기술수준(고급언어 → 5~10배 생산성)
  • 브룩스 법칙: "지체되는 프로젝트에 인력을 더 투입하면 개발은 더 늦어진다"(신규 인력 교육 필요)
  • 보헴: 정상 계획에서 일정은 최대 75%까지만 단축 가능
암기 필수

타당성 3종 = 경제적 · 기술적 · 법적

브룩스 법칙 = 늦은 프로젝트 + 인력 추가 → 더 늦어짐

일정 단축 한계(보헴) = 최대 75%

① LOC (원시 코드 라인 수) 기법

추정 LOC = (낙관치 + 4 × 중간치 + 비관치) ÷ 6
  • 노력(M/M) = 추정 LOC ÷ 1인당 월평균 생산 LOC
  • 개발비용 = M/M × 1인당 월평균 인건비
  • 개발기간 = M/M ÷ 참여 인원
  • 생산성 = LOC ÷ M/M

예) 낙관300·중간600·비관900 → (300+2400+900)/6 = 600 LOC
예) 추정 20,000 LOC, 생산성 2,000 → 노력 10 M/M, 인건비 500만 → 비용 5,000만원, 인원 3명 → 기간 ≈ 3.3개월

한계: 구현(코딩) 단계만 대상 → 분석·설계·테스트 노력 누락 → 보완책 = 개발 단계별 노력 기법

암기 필수

추정 LOC 가중치 = 낙관 1 : 중간 4 : 비관 1 (PERT 베타분포). ÷6 잊지 말 것

⚠️ 일부 슬라이드에 "비관 가중치4"로 적혀 있으나 공식상 ×4는 중간치(가장 그럴듯한 값)가 정답

② COCOMO (Constructive Cost Model)

PM = A × (KDSI)B × EAF  |  TDEV = 2.5 × (PM)β
  • KDSI = 납품 코드(SLOC)를 1,000줄 단위로. 주석·공백·라이브러리 호출 제외 (DSI ÷ 1000)
  • 입력 = KDSI, 출력 = 노력(PM)·개발기간(TDEV)
유형적용 범위ABβ
Organic(단순형)5만 라인 이하, 잘 알려진 응용2.41.050.38
Semi-detached(중간형)30만 라인 이하, OS·DBMS·트랜잭션3.01.120.35
Embedded(내장형)HW 포함 실시간(미사일·신호제어)3.61.200.32
  • B>1 → 규모 커질수록 노력 비선형 증가. 노력·증가율: 단순형 < 중간형 < 내장형
  • EAF = 4개 영역 15개 Cost Driver 승수의 곱. 모두 Nominal이면 1.00. 평가 안 하는 항목은 1.00 처리
  • 4영역: 제품(3)·컴퓨터(4)·인력(5)·프로젝트(3) = 15개
  • 인력 역량↑ → 승수 1보다 작아짐(유리) / TIME·STOR은 1 이상만 / SCED는 U자형(너무 촉박·너무 여유 둘 다 1↑)
  • COCOMO II 3단계: ①애플리케이션 합성 ②초기 설계 ③구조 설계 이후(Post-Architecture, FP→LOC)

예) KDSI=60, 중간형, 신뢰도1.15·복잡도1.30·SW기술1.24·일정1.10 → EAF=1.15×1.30×1.24×1.10 ≈ 2.04
PM = 3.0×601.12×2.04 ≈ 600.18 → TDEV = 2.5×600.180.35 ≈ 23.46개월

암기 필수

PM = A·KDSIB·EAF / TDEV = 2.5·PMβ

유형 A/B/β: Organic 2.4/1.05/0.38 · Semi 3.0/1.12/0.35 · Embedded 3.6/1.20/0.32

EAF = 15개 Cost Driver 승수 곱 / Nominal=1.00

③ Function Point (기능 점수) 기법

LOC와 무관하게 사용자 관점의 기능 개수·복잡도로 규모 측정. 언어·구현·팀에 독립적, 전 생명주기 사용 가능.

구분유형약어의미평균 가중치
데이터내부 논리 파일ILF시스템이 직접 관리(CUD)하는 파일7.5
데이터외부 연계 파일EIF외부가 유지, 참조만 하는 파일5.4
트랜잭션외부 입력EI등록/수정/삭제(ILF 갱신)4.0
트랜잭션외부 출력EO계산·가공·유도 데이터 출력5.2
트랜잭션외부 조회EQ단순 조회(계산 없음, 정렬·페이징 OK)3.9
  • 데이터 기능점수 = ILF×7.5 + EIF×5.4 / 트랜잭션 기능점수 = EI×4.0 + EO×5.2 + EQ×3.9
  • UFP(미조정 기능점수) = 데이터 + 트랜잭션 기능점수 (비기능 요소 제외)
  • 보정 전 개발원가 = UFP × 기능점수당 단가(2025년 605,784원) 구버전 519,203원
  • 보정 후 원가 = 보정 전 × (규모 × 애플리케이션유형 × 언어 × 품질/특성 보정계수)
  • 정규 FP법 = 설계 이후 / 간이 FP법 = 기획·발주 단계(평균 복잡도 사용)

학사관리 예) UFP=158 → 보정 전 = 158×605,784 = 95,713,872원
보정계수 0.65×1.0×0.8×1.0 = 0.52 → 보정 후 ≈ 49,770,823원

암기 필수

5유형 가중치 = ILF 7.5 · EIF 5.4 · EI 4.0 · EO 5.2 · EQ 3.9

분류 4질문: ①데이터/처리? ②(데이터)내가 CUD? → ILF/EIF ③(처리)입력? → EI ④(출력)계산 있나? → EO/EQ

단가 = 605,784원(2025) / UFP = 데이터+트랜잭션 / 간이FP=기획·발주

부록 A · 5유형 분류 함정 (자주 틀림)

  • 물리적 위치 ≠ 기준. 내 DB에 있어도 외부가 관리·동기화만 하면 EIF (예: 우편번호 마스터)
  • 조회 화면이라도 내부에 SUM/AVG 계산이 있으면 EO (예: "보유 적립금 1,250원" 표시)
  • 입력+단순응답이 한 트랜잭션이면 EI 1개로 카운트(주문등록 후 주문번호 표시)
  • SQL: INSERT/UPDATE/DELETE/MERGE → EI · 단순 SELECT/정렬/페이징/JOIN → EQ · COUNT/SUM/AVG/파생컬럼 → EO
  • 같은 데이터라도 측정 대상이 누구냐에 따라 ILF↔EIF가 바뀜(학사 EIF "교수정보" = 인사시스템엔 ILF)

13주차 · 일정 계획 (WBS · CPM · 간트) & 위험 분석

WBS (작업 분할 구조도)

  • 프로젝트를 활동·업무로 세분화하여 계층 구조로 분류, 전체 범위 정의
  • 작업 패키지 = 계층의 최하위 항목
  • 용도: 의사소통 도구, 업무 가시화, 책임·역할 명확, 일정·인력·개발비 산정 기초, 성과측정 기준선

PERT / CPM (네트워크 차트)

약어의미계산
ES가장 빠른 시작앞(시작)에서부터 전진 계산
EF가장 빠른 완료ES + 작업 소요시간
LS가장 늦은 시작뒤(끝)에서부터 후진 계산
LF가장 늦은 완료LS + 작업 소요시간
ST여유시간(Slack)LS − ES (= LF − EF)
  • 임계경로(Critical Path) = 여유시간(ST)이 0인 경로 = 프로젝트 최소 완료기간을 결정
  • 간트차트 = 일정 관리를 위한 바(bar) 형태 도구

위험 관리 4절차

  • ① 위험 식별 → ② 위험 분석(발생가능성·영향력) → ③ 위험 계획 수립(대응방안) → ④ 위험 감시
  • 발생 가능성 척도: 매우낮음(<10%)·낮음(10~25%)·보통(25~50%)·높음(50~75%)·매우높음(>75%)
  • 영향력: 비용·일정 기준 상(20%↑)·중(5~20%)·하(5%↓)
암기 필수

EF = ES + 소요시간 / LF = LS + 소요시간 / ST = LS − ES

임계경로 = 여유(ST)가 0인 경로, 최소 완료기간 결정

작업 패키지 = WBS 최하위 / 간트차트 = 바 형태

위험관리: 식별 → 분석 → 계획 → 감시

Decorator구조상속 대신 객체를 감싸 기능을 동적으로 추가
Facade구조복잡한 서브시스템을 단순 인터페이스(대표창구)로 제공
Proxy구조실제 객체 대신 접근 제어 대리 객체(지연 로딩 등)
Composite구조부분-전체 트리, 단일·복합 객체를 동일하게 다룸
Observer행위상태 변화 시 관련 객체에 자동 통지(이벤트)
Strategy행위알고리즘군을 캡슐화하여 런타임에 교체
Command행위요청을 객체로 캡슐화(실행/취소 Undo 가능)
State행위객체 상태별로 동일 메서드 동작을 다르게 처리
Template Method행위알고리즘 틀은 상위, 세부 단계는 하위 클래스에 위임
Mediator행위객체 간 직접 통신 대신 중재자를 통한 간접 통신
Visitor행위객체 구조와 기능을 분리(구조 변경 없이 기능 추가)
Memento행위객체 상태 저장·복원(Undo)
Iterator행위집합체 요소를 동일 인터페이스로 순차 접근
Interpreter행위간단한 언어의 문법을 클래스화해 해석(SQL·프로토콜)
Chain of Responsibility행위처리 못 하면 다음 처리자에게 자동 전달(체인)

실무 활용 / 효과적 조합

  • 현업 다용: Singleton(연결·Logger), Factory Method(Bean·드라이버), Strategy(인증·정렬), Observer(이벤트), Proxy(AOP·보안)
  • 조합: Factory+Singleton(커넥션 풀), Strategy+Template Method, Observer+Mediator, Command+Memento(Undo/Redo), Composite+Visitor
  • 협업 효과: 역할 분리 명확, OCP(개방-폐쇄) 기반 확장, 병렬 개발, 패턴명만으로 설계 의도 전달
